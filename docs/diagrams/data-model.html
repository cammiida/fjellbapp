Cabin Renovation Tracker — Data Model

The agreed initial schema. Everything is scoped to one Project; people and companies share one Party table (roles, not types); IDs are client-generated UUIDs. v3 folds in the grilling-session refinements: a polymorphic Expense Line (one receipt, many items), a Tool Availability table (one tool, many hold windows), a mandatory polymorphic target on Note, and a model-wide provenance + Audit Log convention. Domain clusters, deferred concepts, and key decisions follow the diagram.

Reading the people corner. PARTY holds both people and organizations (kind). employs is the self-referential employer_party_id — a person points to their org (app-validated as kind = organization). A MEMBER is a login that references its person PARTY. Work is assigned through TASK_ASSIGNMENT (M2M) to any party — you, a contact, or a whole shop. PARTY_ROLE tags a party as contractor / supplier / collaborator, so suppliers are just parties (see MATERIAL.supplier_party_id). ATTACHMENT_LINK, EXPENSE_LINE and NOTE's target all use the same polymorphic entity_type + entity_id shape — the may target edges shown are representative, not exhaustive. Two model-wide conventions are omitted from the boxes for clarity, exactly as project_id is: the provenance columns (created_at / created_by / updated_at / updated_by) on every aggregate root, and the AUDIT_LOG row written by every mutation (capture-now, view-later).
Domain clusters

People & parties core

Project, Party (person/org), Party Role, Member, Membership, Task Assignment.

One Party table for everyone; roles not types; a Member is a login bound to a person-party. Work assigns M2M to any party.

The build core

Phase, Task, Task Dependency, Task Material, Task Tool, Material, Tool, Milestone.

Tasks sit in one Phase and form a finish-to-start DAG, and assign to parties. A Task also needs materials (with quantity) and tools — feeding the readiness check.

Money core

Budget Category, Budget (target per category), Expense, Expense Line.

Estimates are forecasts; the single Expense ledger is the source of truth. One Expense itemizes into many Expense Lines, so one shop receipt can pay for many Materials (and labour invoices target a Party).

Documentation core

Attachment, Attachment Link, Note.

One mechanism for photos + documents in R2; the Build log is a view of all images ordered by taken_at.

Non-obvious modelling decisions

One Party table, roles not types

Person & Organization share one table (kind); name/kind are columns, the rest is metadata JSONB. Contractor/Supplier are roles a party plays — one company can be both. No duplication.

Assignment is M2M to Party

A Task assigns to one or more parties via task_assignment — a Member, a specific contact, or a whole two-person shop. The expensive-to-retrofit boundary, so decided up front.

Member ≠ Person

Member = can log in (credentials, app role); Party = who they are. Member references its person-party 1:1 — a user is a person without storing identity twice.

A Task needs tasks, materials & tools

Three task_* link tables — task_dependency (other tasks), task_material (+ quantity), task_tool. Together they drive readiness in two tenses: ready now (blockers done + materials received + a tool window covers the dates) and at risk (a material's expected_arrival lands after the task's start). No auto-reschedule.

One Tool, many hold windows

The Tool row is stable identity (so "buy the saw once" never double-counts); tool_availability holds zero-or-many windows, so the same excavator rented in spring and autumn is two windows on one row, not two tools.

Material: bulk vs. fixture

Bulk = quantity + unit (cladding by m²); fixture = one row per item (each window), specs in JSONB. A window is a Material; expected_arrival feeds the at-risk signal.

Provenance + append-only audit

Every aggregate root carries created_at/by + updated_at/by; every mutation also writes an audit_log row (who, when, before/after) via one audited write-path. Captured from day one, viewer deferred. See ADR-0006.

Milestone is its own time axis

A target date that spans phases (weathertight = roof + windows + sealing), orthogonal to the work breakdown — the time-axis twin of Budget Category on money. Fulfilled by tasks (M2M); on-track is derived, only achieved_at is stored & manual.

Polymorphic links, not nullable FKs

attachment_link, expense_line and Note's target share one entity_type+entity_id shape. One photo documents a Task and a Material; deleting an entity drops its links, never the file.

Deferred — concept agreed, seam left, no rework to add

Property above Project deferred

A durable Property (the cabin) will sit above Project — one cabin, many projects (renovation, remodels, annual maintenance). Durable things (rooms, requirements, owned tools, parties, access, build log) move up; effort-scoped things stay on Project.

Seam: inserting a parent above a single-Project root is a cheap, unambiguous retrofit; build deferred durable concepts property_id-scoped from birth. See ADR-0007.

Affiliation join table deferred

Today a person has one employer via employer_party_id (self-FK).

Seam: promote to affiliation(person, org, title, is_primary_contact) for many orgs + roles per person.

Room deferred

Per-room rollups may never be needed for one cabin.

Seam: nullable room_id on Task / Material / Expense, optional backfill.

Requirement deferred

The durable bound on a slot (“U-value ≤ 1.0”) that outlives any fixture filling it.

Seam: Requirement entity a fixture satisfies; actual vs. bound kept distinct.

Household inventory + Wishlist deferred

“What the cabin has & wants” — separate from build supplies.

Seam: own entity reusing the have/want status pattern.

Permits & inspections deferred

Dates + approval that gate work — an external dependency.

Seam: entity that blocks Tasks like a Dependency.

Labour time entries deferred

Hour-level tracking. Today: quotes in, invoices out.

Seam: optional TimeEntry rows (hours × rate); Expense link unchanged.

Offline · iCal · Google sync deferred

Light-offline reads/outbox, calendar export, two-way sync.

Seam: client UUIDs + idempotent writes already in place; sync maps by stable IDs.
